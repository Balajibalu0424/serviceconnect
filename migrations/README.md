# Database Migrations

This directory holds versioned SQL migrations generated by `drizzle-kit generate`.

## Workflow

**Before** this folder existed, schema changes were deployed via `drizzle-kit push`
(direct DDL diff against the live DB). That has no audit trail and no way to roll
back. Versioned migrations replace that workflow.

### 1. Modify `shared/schema.ts`
Make the schema change in code as before.

### 2. Generate a migration
```bash
npm run db:generate
```
This diffs the current schema against the last known state and writes a new
timestamped `.sql` file (plus a `meta/_journal.json` entry) into this folder.

Review the generated SQL. If it's destructive (dropping columns, narrowing
types) add any explicit data-migration statements by hand.

### 3. Apply migrations
```bash
npm run db:migrate
```
This runs all pending migrations in order. Safe to run multiple times — drizzle
tracks which have already been applied.

### 4. Commit
Commit both the edited `shared/schema.ts` **and** the new migration file in the
same PR. Reviewers check both.

## Legacy fallback

`npm run db:push:force` still exists for emergency schema syncs in
non-production environments, but must not be used in production.

## Baseline

The first migration in this repo represents the schema snapshot as of
ServiceConnect Session 13 (P0/P1 launch-readiness pass). Generate it once with
`npm run db:generate` against a fresh database before running the app for the
first time under the new workflow.
